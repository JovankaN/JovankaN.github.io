What is TDD?

This is a graphical example of the TDD cycle :

    • Test-driven Development(TDD) is a software developmenet process that relies on the repetition of a very short development cycle :

    First the developer writes an (initially falling) automated test case that defines a desired improvement or new function.We write the test in whatever testing framework is appropriate to the software environment, it could be also a modified version of an existing test.

    Second produces the minimum amount of code to pass that or that the new test does not mistakenly pass without requiring any new code, and that the required feature does not already exist.This step also tests the test itself, in the negative: it rules out the possibility that the new test always passes, and therefore is worthless. The new test should also fail for the expected reason.

    Third    step is to write some code that causes the test to pass. The new code written at this stage is not perfect and may, for example, pass the test in an inelegant way

    FourthWe run all the tests so that the developer can be confident that all the tests pass.If they don't , the code needs to be adjusted so it will pass al the tests.

    Fifth refactors the new code to acceptable standards and the fourth step is to run your tests again. If they fail you need to update your functional code and retest. Once the tests pass the next step is to start over (you may first need to refactor any duplication out of your design as needed, turning Test-First Development(TFD) into TDD).

    Some of the developers refered to this simple formula for better understanding :

    TDD = Refactoring + TFD;


    There are many more interesting examples on TDD and the way it works, so if you are still interested in finding more about it visit : http://www.agiledata.org/essays/tdd.html ,
